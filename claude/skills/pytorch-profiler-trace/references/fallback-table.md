# fallback-table.md — torch.profiler failure scenarios

| Situation | Symptom | Fallback 1 | Fallback 2 | Stop if |
|---|---|---|---|---|
| Profiler overhead distorts timing | Profiled run is noticeably slower; results look different from unproflied | Drop `record_shapes`, `with_stack`, `profile_memory`; use `activities=[CUDA]` only | Shorten `active=1` and profile a single step; compare wall time with/without profiler | Overhead persists even with minimal config — use `nsys-trace-profiler` for lower-overhead system trace |
| Trace too large / Chrome unresponsive | `trace.json` is hundreds of MB; browser freezes on load | Reduce `active=1`; use `row_limit` in `key_averages()` instead of Chrome trace | Use `nsys stats` approach: `key_averages().table()` only; skip Chrome export | Trace is irreducibly large — open with Perfetto UI (handles larger traces than Chrome) |
| CUDA Graph activity not visible at node level | Graph launch appears as single opaque block; no per-node timing | Confirm `with_flops=False`, add `record_shapes=False`; retry | Switch to `nsys-trace-profiler` with `--cuda-graph-trace=node` — this is not visible in torch.profiler | torch.profiler cannot show CUDA Graph node-level detail; nsys is required |
| NCCL / comm not visibly overlapping compute | Timeline shows comm and compute sequential; unclear if overlap is real | Export Chrome trace and zoom into stream rows; NCCL kernels appear as `ncclKernel_*` CUDA events | Use `nsys-trace-profiler` for system-level overlap view with explicit stream rows | NCCL kernels are present but timing granularity is insufficient — nsys gives cleaner stream-level view |
| Dataloader overhead not visible | GPU shows idle gaps but no obvious CPU op accounts for them | Profile with `activities=[CPU, CUDA]` and `with_stack=True` to trace Python-side overhead; look for `__getitem__` or collate ops | Compare traces: one with `num_workers=0` (loader in main process) vs. normal; gap difference = loader overhead | Loader overhead confirmed; reduce `num_workers`, increase prefetch, or use `persistent_workers=True` |
| CPU is idle but GPU is also idle | `key_averages()` shows low both CPU and CUDA time; wall clock is slow | Look for Python-level overhead not captured by profiler (`with_stack=True` to find culprit); check for synchronization (`cuda.synchronize`) | Add timing around individual components (data load, forward, backward, optimizer) to find the wall-clock gap | Gap is in code outside the profiled region — widen the profiled window |
| All ops look fast but wall time is slow | `key_averages()` totals don't add up to wall time; profiling window captures fast ops but overall throughput is low | Check whether profiling is capturing only a portion of the iteration (wrong schedule); extend `active` window | Look for Python overhead between steps: GC, logging, checkpoint saves, or other Python work not wrapped in `record_function` | Wall time includes non-profiled code — identify by timing each section manually |
| Kernel names missing or garbled | CUDA events show `<unknown>` or address-only names; no meaningful kernel names | Ensure binary/model is not compiled with aggressive symbol stripping; use `torch.compile` with `mode="default"` not `"max-autotune"` for cleaner names | Use `nsys-trace-profiler` — `cuobjdump --list-symbols` can resolve names from the cubin | Kernels are from a stripped third-party library — work with timing data only |
