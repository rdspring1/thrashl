# fallback-table.md — nsys trace failure scenarios

| Situation | Symptom | Fallback 1 | Fallback 2 | Stop if |
|---|---|---|---|---|
| Trace file too large | `.nsys-rep` is many GB; takes too long to open or generate | Add `--duration 10` to cap trace length; or add `--delay` to skip startup | Use `--capture-range=cudaProfilerApi` — requires adding `cudaProfilerStart()`/`Stop()` in source | File is still too large after scoping; summarize with `nsys stats` only |
| Graph activity not visible | Timeline shows no graph nodes; kernels appear as individual launches | Add `--cuda-graph-trace=node` to expose node-level execution | Confirm graphs are actually executing (not just captured): add NVTX annotation around `cudaGraphLaunch()` and check if it appears | Node trace is present but shows only capture phase — graph may not be replaying |
| CPU overhead not visible | Timeline shows GPU activity but no CPU API calls or thread rows | Add `osrt` to `-t` (e.g., `-t cuda,nvtx,osrt`) | Add `--sample=process-tree --backtrace=fp` for CPU call stacks | CPU rows are visible but launch latency looks correct — no further action needed |
| Trace misses event of interest | Run completes before trace captures the relevant phase | Add `--delay <N>` to skip startup, or increase run iterations | Use `--capture-range=cudaProfilerApi` scoped around the relevant phase | Phase cannot be reliably scoped — report what was captured and what is missing |
| Tracing overhead distorts results | Kernel timing shifts significantly vs. baseline; CPU rows dominate | Drop `osrt` from `-t`; use only `cuda,nvtx` | Drop NVTX if app has dense annotations; use `-t cuda` only | With minimal `-t cuda` trace, overhead is acceptable — proceed with results as-is |
| NCCL / comm activity missing | No NCCL kernels visible; only compute kernels shown | Add `nccl` to `-t`; confirm nsys ≥ 2022.3 and NCCL ≥ 2.12 | Set `NCCL_DEBUG=INFO` and check whether NCCL is actually active during the trace window | NCCL kernels are present but labeled as `ncclKernel_*` with no further breakdown — this is expected |
| Kernel names garbled or missing | Kernels appear as `[CUDA memcpy]` or hex addresses; no function names | Ensure binary is not stripped; recompile with `-g` or `-lineinfo` if needed | Use `nsys stats --report cuda_kern_sum` — names come from the binary's symbol table | Symbols are genuinely unavailable (third-party binary); work with addresses and durations only |
| Graph capture visible but replay not | Timeline shows long graph capture phase but no subsequent graph node launches | Confirm `cudaGraphLaunch()` is called after capture; add NVTX around the launch | Use `--cuda-graph-trace=node` if not already set; check whether the graph is being updated and re-captured every iteration | If capture and launch are interleaved every iteration, graph is not amortizing launch cost — this is the finding |
